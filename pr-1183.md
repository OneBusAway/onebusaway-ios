## Important Issues (3 found)

**`MapViewModel.settingsURL` is dead code and its doc comment contradicts itself**
`OBAKit/ViewModels/MapViewModel/MapViewModel.swift:189-197`

`settingsURL` is defined but never referenced — `MapPanelRootView.openSettings()` (line 202-205) reads `UIApplication.openSettingsURLString` directly. The comment argues the wrapper exists "so SwiftUI call sites don't need to depend on `UIApplication.openSettingsURLString` directly," but the one SwiftUI call site does exactly that. Either:

- Delete `settingsURL` and its comment (my preference — the direct reach into `UIApplication` in `openSettings()` is fine; UIKit is already imported in `MapPanelRootView`), or
- Actually use it from `MapPanelRootView.openSettings()` and drop the `UIApplication.openSettingsURLString` reference.

The force-unwrap comment ("`app-settings:` — the same string UIKit's constant resolves to on all supported iOS versions") is also a promise the code shouldn't be making — that's an internal Apple implementation detail. If we keep the helper, hand it `UIApplication.openSettingsURLString` and stop hard-coding the scheme.

**`confirmationDialog` renders redundant dismissal buttons**
`OBAKit/Sheet/Root/Controls/MapPermissionDialog.swift:98-137`

Now that `.cancel` role is stripped from "Keep Location Off" / "Keep Precise Location Off" (so their labels render), SwiftUI auto-injects a system Cancel button because no `.cancel`-role button is present. The user sees three rows for `.locationServicesOff`:

- Turn On in Settings
- Keep Location Off (no-op)
- Cancel (system auto-added, no-op)

…and four for `.impreciseLocation`. Two dismiss-only rows read as a bug. Options, roughly in order of preference:

1. Use `.alert(...)` instead of `.confirmationDialog(...)`. Alerts don't auto-add Cancel and let you keep the "Keep Location Off" affordance as the meaningful opt-out. This matches the UIKit `didTapMapStatus` alert too.
2. Give "Keep Location Off" the `.cancel` role and pass the localized label as the title — worth testing on iOS 18 and 26 in case the label-swallowing behavior you hit earlier was version-specific.
3. Drop "Keep Location Off" entirely and rely on the system Cancel. Loses the explicit opt-out phrasing, which I don't love.

I'd take (1). The label copy in the UIKit alert was the whole point of the earlier fix, and `.alert` preserves it without the phantom Cancel row.

**Two unreachable branches in `handlePermissionTap`**
`OBAKit/Sheet/Root/MapPanelRootView.swift:135-146`

`MapStatusPill.body` returns nothing for `.hidden` (via `Display?`) — the pill is not in the view tree, so no tap can fire. And for `.zoomInForStops`, `MapStatusPill.handleTap()` routes to `onZoomInForStops`, never to `onPermissionTap`. The `.hidden` `assertionFailure` and the `.zoomInForStops: break` are both dead.

Collapse to a single non-exhaustive switch or replace with a guard:

```swift
private func handlePermissionTap(_ state: MapViewModel.TopPillState) {
    guard case .notDetermined = state, case .locationServicesOff = state, case .impreciseLocation = state else { return }
    permissionDialogState = state
}
```

…or, simpler and just as safe:

```swift
private func handlePermissionTap(_ state: MapViewModel.TopPillState) {
    // Only permission states drive the dialog; other cases are routed elsewhere in MapStatusPill.
    switch state {
    case .notDetermined, .locationServicesOff, .impreciseLocation:
        permissionDialogState = state
    case .hidden, .zoomInForStops:
        break
    }
}
```

The dead-code branches will eventually invite someone to "fix" them by wiring behavior in, which will confuse the pill's actual routing.

## Suggestions (5 found)

**`.mapStyle(.standard)` isn't parity with UIKit's `mutedStandard`**
`OBAKit/Sheet/Root/MapPanelRootView.swift:54`

The `MapBaseType` doc comment explicitly says "UIKit maps `.standard` → `MKMapType.mutedStandard`". The SwiftUI Map is using the bright `.standard` style instead. `MapStyle.standard(emphasis: .muted)` is the closer equivalent. Not a blocker; worth trying to see if it matches the UIKit look side-by-side.

**Magic 48-pt offset for the weather button under the pill**
`OBAKit/Sheet/Root/MapPanelRootView.swift:117`

`.padding(.top, mapViewModel.topPillState == .hidden ? 0 : 48)` hardcodes the pill's height. If Dynamic Type or accessibility text sizes push the pill taller, they'll overlap again. Consider a `PreferenceKey` on `MapStatusPill` (or reading its size via `onGeometryChange`) and using the measured height + a small gap. Same category as the existing detent TODO — leave a comment referencing it if you don't want to fix it now.

**`initialMapType` snapshotted in `init` never re-syncs from `mapRegionManager`**
`OBAKit/Sheet/Root/MapPanelRootView.swift:44`

If the UIKit path (or any other consumer) mutates `application.mapRegionManager.userSelectedMapType` while the SwiftUI root is mounted, the SwiftUI `Map` won't reflect it. During the migration both paths can be alive across a scene lifecycle. Two options: subscribe to `$userSelectedMapType` from the VM and mirror into `mapType`, or push the initial-derivation into `MapViewModel.init` (via a factory param) and treat `$mapType` as the sole source of truth. Not urgent; worth logging.

**Test coverage gaps**
`OBAKitTests/ViewModels/MapViewModelTests.swift:471-540`

- `zoomLevelForCurrentLocation()` — only the full-accuracy branch (17) is tested. The reduced-accuracy branch (11) is a one-line conditional; it deserves a companion test that mocks `accuracyAuthorization == .reducedAccuracy`.
- `TopPillState.impreciseLocation` — the "authorized + reduced accuracy → `.impreciseLocation`" branch isn't exercised. That's the trickiest transition in `topPillState`; the others are more straightforward.

- The `for _ in 0..<5 { await Task.yield() }` pattern is a smell (waiting for the `Task { @MainActor in … }` hop inside `locationService(_:authorizationStatusChanged:)`). If the delegate ever moves off `nonisolated`, the tests silently pass with fewer hops. Consider adding a small `awaitPublishedChange` helper or await a `for await` over `$locationAuthStatus.values.first(where:)`.

**Icon style inconsistency inside the cluster**
`OBAKit/Sheet/Root/Controls/CurrentLocationButton.swift:24-27`, `OBAKit/Sheet/Root/Controls/MapTypeButton.swift:22-25`

`CurrentLocationButton` renders `Icons.nearMe` (asset image with `.template`), while `MapTypeButton` uses SF Symbols directly. Both work, but they sit ~8pt apart in the same cluster. If `Icons.nearMe` matches the UIKit control we're paralleling, keep it; otherwise consider using `Image(systemName: "location")` for a consistent glyph weight/scale next to the map-type toggle.

## Strengths

- The `MapPermissionDialog` viewmodifier + `TopPillState?` binding is a clean pattern — presentation state is out of the VM, the pill stays display-only, and the sheet-vs-map-layer question is settled in one place. Doc comment says exactly why.
- `MapStatusPill`'s internal `Display?` collapsing `.hidden` to `nil` is a much nicer model than nested `if state != .hidden` at every call site.
- Extracting `zoomInForStopsSpan` and `zoomLevelForCurrentLocation()` into `MapViewModel` and updating `MapViewController` to call them — that's exactly the boring, quiet cleanup that keeps parallel UIKit/SwiftUI paths honest.
- `MKCoordinateRegion(centeredOn:zoomLevel:mapSize:)` reuses `MapHelpers.coordinateSpanFrom(...)`, so the SwiftUI "center on user" behaves identically to `MKMapView.setCenterCoordinate(zoomLevel:)`. Good instinct.
- `liquidGlassButtonStyle(borderShape:fallbackShape:)` is a nice, minimal capability abstraction — iOS 26 gets the real thing, older systems get a plausible material fallback, callers don't spell out an `if #available`.
- Small fix commits (dialog dismissing sheet, `.cancel` label stripping, weather-button overlap) each have clear, single-line subjects that explain the concrete cause. Easy to review in isolation.
- `MapControlsCluster` is genuinely one-purpose: a vertical stack of two floating buttons, no state, no side effects. A future contributor can drop in a third button without rewiring anything.
- Fixture-driven `TopPillState` transitions are covered where they matter (zoom-warning precedence, `.denied` → `.locationServicesOff`, `.notDetermined`). See the suggestion above for the two branches that aren't.
